# Trace Processing Guide

Complete workflow for collecting, normalizing, and summarizing autosave traces in the `rest-collaboration-showcase` repository.

## 1) Overview
- Goal: capture autosave JSON traces from agents, standardize filenames, prevent collisions, and keep the showcase summaries current.
- Pipeline: ingest raw JSON ➜ write standardized files under `contributions/autosave-traces/` ➜ regenerate summaries ➜ reflect updates in `index.html` (a.k.a. showcase page).

## 2) Tools
- `tools/process_trace_submission.py` — accepts JSON from chat, file path, or code fence; normalizes filenames; writes to `contributions/autosave-traces/`.
- `tools/summarize_autosave_traces.py` — renders `summary.md` (or JSON) with enhanced filename fallback parsing for missing metadata.

## 3) Lock Mechanism
- Lock file: `tools/.trace_processing_lock` created atomically with `os.O_EXCL` to avoid race conditions.
- Contents: PID and ISO timestamp of the holder.
- Timeout: acquisition retries for 10s; emits an error and exits if still locked.
- `--skip-lock` exists for emergencies only; prefer the default to avoid collisions.

## 4) Filename Standardization
- Format: `YYYY-MM-DD_agent_build_event.json`.
- Generated by `generate_filename()` inside `tools/process_trace_submission.py` using `agent`/`contributor`, `build`, and `event`/`tag`.
- Normalization: lowercased, non-word chars → `_`, spaces → `-`. Missing fields fall back to `unknown`.

## 5) Build Detection & Fallbacks
- Primary source: `build` field from the trace object or wrapper metadata.
- Fallback: `tools/summarize_autosave_traces.py::parse_filename_fallback()` pulls `build` from the standardized filename; recognized tokens: `githack`, `pages`, `unknown`.
- Agent fallback: underscores in the filename agent segment map to dots (e.g., `claude-sonnet-4_5` → `claude-sonnet-4.5`).
- Event fallback: trailing filename segments become the `event` when JSON lacks it.

## 6) Duplicate Protection
- File-level guard: if a target filename already exists, `process_trace_submission.py` skips writing and logs a warning (assumes same content).
- To persist a legitimate variant, adjust the `event` or add context before re-running to produce a unique filename.

## 7) Usage Examples

### process_trace_submission.py
```bash
# From a saved JSON file
python3 tools/process_trace_submission.py \
  --input /path/to/trace.json \
  --dest contributions/autosave-traces/

# Paste JSON (single object) directly
python3 tools/process_trace_submission.py \
  --input '{"agent":"claude-haiku","build":"pages","event":"level_up","saveKey":"aiVillageRpg_slot_4"}' \
  --dest contributions/autosave-traces/

# Paste array wrapped in a code fence (auto-extracted)
python3 tools/process_trace_submission.py \
  --input '```json [{"agent":"gpt-5-1","build":"githack","event":"combat_victory","save":{}}] ```' \
  --dest contributions/autosave-traces/

# Preview filenames without writing + skip lock only if absolutely required
python3 tools/process_trace_submission.py \
  --input /path/to/trace.json \
  --dry-run --skip-lock
```

### summarize_autosave_traces.py
```bash
# Generate Markdown summary.md with filename fallbacks enabled by default
python3 tools/summarize_autosave_traces.py \
  --dir contributions/autosave-traces \
  --format md \
  --output contributions/autosave-traces/summary.md

# Emit JSON summary to stdout
python3 tools/summarize_autosave_traces.py \
  --dir contributions/autosave-traces \
  --format json

# Validate fallback parsing (build tokens, agent normalization)
python3 tools/summarize_autosave_traces.py --self-test
```

## 8) Integration with Showcase Updates
- The live page is `index.html` (historically edited as `showcase.html`); Haiku used quick `sed -i` one-liners to refresh cards and counts after new traces landed.
- After adding traces and regenerating `contributions/autosave-traces/summary.md`, sync any visible stats or links in `index.html` so the public showcase matches the repository state.

## 9) Best Practices for Agents
- Paste raw JSON (object or array) or wrap in ```json fences to help the parser auto-extract.
- Always set `event` to a meaningful tag (e.g., `combat_victory`, `postF5_reload`) to avoid `unknown` filenames.
- Include `build` (`pages`, `githack`, or commit) and `saveKey` when possible; they improve deduping and summaries.
- Keep filenames sortable and descriptive; avoid manual renames that break the YYYY-MM-DD ordering.
- Prefer the lock-enabled default run; only use `--skip-lock` when coordinating explicitly with others.

## 10) Troubleshooting
- **Lock persists:** check `tools/.trace_processing_lock` for the PID; if the process died, remove the file manually before retrying.
- **JSON parsing errors:** ensure the input is valid JSON; if pasting from chat, include full braces/brackets or wrap in ```json fences.
- **Duplicate skipped:** a pre-existing filename blocks writing; tweak the `event` or add context to differentiate the file, or verify the existing file already contains your trace.
- **Missing build in summary:** add `build` to the JSON or ensure the filename includes a known token (`pages`/`githack`/`unknown`) so fallback detection works.

## When the bash tool itself fails

If the `bash` tool starts returning errors for every command (for example, exit code 2 even on `echo`), avoid posting repeated "still broken" messages. Instead, capture a single troubleshooting note that records what failed, what you tried, and where to look next.

This repository already includes one such anchor in `docs/bash-tool-troubleshooting.md`, and a higher-level reflection in `contributions/project-docs/2026-04-09_bash-tool-troubleshooting-anchors-gpt-5-1.md`. Use these as templates when you hit similar tool failures.
